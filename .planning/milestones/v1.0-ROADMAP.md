# Milestone v1.0: MVP

**Status:** SHIPPED 2026-02-02
**Phases:** 1-7
**Total Plans:** 17

## Overview

This roadmap rebuilt the scholardoc-ocr hybrid OCR pipeline from the ground up to fix critical architectural problems while preserving its core two-phase strategy (Tesseract-first, Surya fallback). The rebuild followed a bottom-up dependency order: establish clean data structures and contracts, enhance quality analysis with multi-signal scoring, extract OCR backends into testable modules, fix engine orchestration with per-file batching, wrap everything in a thin CLI presentation layer, and add MCP server integration.

## Milestone Summary

**Key Decisions:**

- Rearchitect rather than patch (critical bugs + architectural issues compound)
- Per-file Surya batching over monolithic combined PDF (index mapping too brittle)
- Library + CLI design (enables programmatic use and testability)
- Three-tier dictionary scoring for quality analysis
- Composite weights garbled:0.4 dictionary:0.3 confidence:0.3
- Function module pattern for OCR backends (not classes)
- PDFProcessor is PDF-manipulation-only (OCR methods extracted)
- Lazy surya import in run_pipeline to avoid heavy ML deps
- CLI as thin presentation wrapper
- MCP server with lazy imports; page_range via temp file

**Issues Resolved:**

- BUG-01: Surya OCR results were discarded, never written to output
- BUG-02: Surya batch output extraction re-read original bad text
- MCP output_path missing from FileResult (Phase 7 gap closure)
- Pre-existing test_callbacks.py import of removed ExtendedResult

**Technical Debt Incurred:**

- confidence.py imports pytesseract at module level (not lazy like surya.py)
- quality.py line 42: Levinas comment vestige
- quality.py lines 25-26: Pre-existing ruff E501 violations
